**Human explanation pack**
- *Issue:* `Counts` accepts hex/int/`0b` keys and documents `memory_slots` as optional, but mixed-width keys without width metadata produced inconsistent-length bitstrings, so `marginal_counts` silently returned the unmarginalized counts.
- *Expected behavior:* integer-like keys should share a consistent bit width before APIs operate on bit positions.
- *Root cause:* `Counts.__init__` formatted integer-like keys with no known width, leaving smaller values unpadded.
- *Fix:* when `memory_slots` is absent, use `creg_sizes` if present, else pad to the max observed width and warn about ambiguity; bitstring/dit inputs are untouched.
- *Tests:* main regression proves `{"0x0":50,"0x3":30}` → `{"00":50,"11":30}` and `marginal_counts(...,[0])` → `{"0":50,"1":30}`; an adjacent test proves `creg_sizes` alone supplies the width.
